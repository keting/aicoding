# 通用 Issue 或 Discussion 提案 Prompt 模板

## 适用场景

对一个开源项目做开放式待办事项盘点，系统分析 roadmap、代码、文档、已有 issues、discussions、用户反馈、同类项目和外部资料，并根据事项成熟度分别建议创建 issue 或继续发起 / 保留 discussion。

此模板比“通用 Issue 提案 Prompt 模板”更开放，适合项目方向尚未完全收敛、维护者希望同时发现可执行任务和需要社区共识的问题时使用。

## 建议提供的上下文

- 项目仓库链接（公开仓库或 agent 有权限访问的仓库）
- 当前关注阶段或主题（例如公开发布前、社区化阶段、文档体验、插件机制、安全治理）
- 项目类型（默认填写“软件开发类”，也可按需改为工具库 / Web 系统 / Agent 框架 / 文档型 / CLI / 其他）
- 期望产出的候选事项数量上限
- agent 是否具备 web 检索、GitHub issues 和 discussions 访问能力

## 使用方式

使用前只需要替换 Prompt 开头变量区中的以下值：

- `REPO_URL`：开源项目仓库链接
- `PROJECT_TYPE`：项目类型，默认填写“软件开发类”
- `FOCUS_SCOPE`：本次关注范围，默认可填写“下一阶段待办事项”
- `CANDIDATE_COUNT`：候选事项数量上限
- `HAS_GITHUB_META_ACCESS`：agent 是否能访问 GitHub issues / discussions，默认填写“是”
- `HAS_WEB_ACCESS`：agent 是否具备 web 检索能力，默认填写“是”

## Prompt

```text
# 变量区：只需要修改本区，其余正文不要手动替换

REPO_URL = <<替换:开源项目链接,例如 https://github.com/org/repo>>
PROJECT_TYPE = 软件开发类
FOCUS_SCOPE = <<替换:本次关注范围,例如 下一阶段待办事项 / 公开发布前 / 社区化阶段 / 文档体验 / 插件机制>>
CANDIDATE_COUNT = <<替换:候选事项数量上限,例如 10-15;合格不足时宁少勿凑>>
HAS_GITHUB_META_ACCESS = 是
HAS_WEB_ACCESS = 是
OUTPUT_LANGUAGE = 中文

# 任务

你是一名经验丰富的开源项目 maintainer 兼 issue triager。

请围绕 REPO_URL 指向的开源项目，分析 FOCUS_SCOPE 范围内值得推进的待办事项，并根据成熟度将其分成两类：

1. Issue 候选：问题明确、边界清晰、可执行、可验收，适合直接创建 issue。
2. Discussion 候选：方向有价值，但还需要方案讨论、需求澄清、社区共识或维护者决策，适合创建或继续 discussion。

本任务允许适度发散和 brainstorm，但最终要回到证据、成熟度和协作载体判断；目标是帮助维护者把待做事项放到合适的 issue 或 discussion 中管理。

不要直接创建 GitHub issue 或 discussion，只输出建议和可复制草案。

# 工作原则

1. 不要只基于 README 或项目名判断，要结合可访问到的代码、文档、roadmap、现有 issues、discussions、release 信息，以及在 HAS_WEB_ACCESS = 是时可访问的同类项目和外部资料。
2. 每个候选事项必须说明依据；纯推断可以保留，但要标注为低证据强度，除非边界和验收方式已经非常清楚。
3. 如果 HAS_GITHUB_META_ACCESS = 否，跳过 GitHub issues / discussions 检索，并明确标注“未完成去重，需要人工复核”。
4. 如果 HAS_WEB_ACCESS = 否，跳过同类项目和外部资料检索，不要编造外部来源。
5. 不要为了达到 CANDIDATE_COUNT 凑数；合格不足时少输出。
6. 无法访问 REPO_URL 时停止分析并报告原因，不要凭项目名编造内容。
7. 关键不确定性不阻碍初步判断时，写出假设并继续；只有无法继续判断时才向用户提问。

# 分析方法

请按以下顺序执行，并在最终输出中集中说明实际访问到的信息来源、未访问到的信息来源及其影响。

## 1. Roadmap 与文档

阅读 README、ROADMAP、TODO、milestone、release plan、CHANGELOG、docs、CONTRIBUTING 等资料，识别：

- 已明确列出的后续事项；
- 已承诺但尚未完成的功能、文档、测试或治理工作；
- 粒度过大、需要拆解的事项；
- 与 FOCUS_SCOPE 强相关的任务；
- 过时、重复或暂不适合推进的事项。

## 2. 代码与工程现状

浏览核心目录，理解模块划分和实现现状，重点识别：

- 当前能力中的明显缺口；
- TODO / FIXME / hack / temporary / deprecated 等信号；
- 错误处理、配置、权限、日志、状态管理、测试、安装、部署、开发体验方面的断点；
- 可以独立完成的小型维护、重构、测试或文档任务；
- 需要先讨论架构、产品方向或兼容性策略的事项。

## 3. Issues 与 Discussions 去重

在 HAS_GITHUB_META_ACCESS = 是时，检索 open issues、近期 closed issues、milestones 和 discussions：

- 已有近似 issue：引用编号，并说明建议是补充、拆分、合并还是放弃；
- 已有近似 discussion：引用链接或标题，并说明建议是继续讨论、沉淀为 issue，还是合并重复讨论；
- 用户反馈、使用问题、功能建议：判断应进入 issue 还是 discussion；
- 无法完成去重时，不得声称没有重复项。

## 4. 同类项目与外部资料

仅在 HAS_WEB_ACCESS = 是时执行。

选择少量高度相关的同类项目、官方文档、最佳实践、安全维护指南或社区规范。说明其相关性，并只提炼适合该项目当前阶段参考的做法。不要机械复制对方功能，也不要强行列出低相关来源。

# 项目类型适配

根据 PROJECT_TYPE 调整侧重，不强求每类事项都覆盖：

- 工具库 / SDK：API 设计、向后兼容、版本策略、示例、测试覆盖、发布流程；
- Web 系统：用户路径、权限、数据一致性、部署运维、UI / UX、可观测性；
- Agent 框架：核心抽象、工具 / 插件机制、执行隔离、上下文管理、调试能力、模型集成；
- 文档型：信息结构、导航、链接一致性、版本管理、可贡献性、示例质量；
- CLI：命令边界、参数设计、跨平台、错误信息、安装升级、shell 集成；
- 软件开发类：根据仓库实际形态综合判断，不强行套用某一类。

# 分流标准

## 建议进入 Issue 的条件

同时满足以下条件，才标为“建议创建 issue”：

1. 问题明确：一句话能说清楚要解决什么。
2. 边界清晰：能指出涉及的模块、目录、文档或流程。
3. 可执行：可以交给一个人或一个 coding agent 独立推进。
4. 可验收：完成后能通过客观标准判断是否达成。
5. 粒度适中：不是大型项目，也不是没有独立价值的小改动。
6. 证据明确：至少有一个具体依据。
7. 不依赖重大未决讨论：不需要先决定路线、方案或社区共识。
8. 不重复：未被已有 open issue 覆盖；如果无法查重，必须标注“需人工查重”。

## 建议进入 Discussion 的条件

满足以下任一情况，优先标为“建议创建 / 继续 discussion”：

1. 方向有价值，但目标、用户场景或成功标准还不清楚。
2. 需要维护者在多个方案之间做取舍。
3. 涉及架构路线、兼容性策略、治理规则、社区规范或长期维护承诺。
4. 可能影响较多用户或已有 API，需要收集反馈。
5. 目前只有外部借鉴或零散反馈，缺少本项目内的明确证据。
6. 已有 discussion 正在讨论，不应绕过讨论直接创建 issue。

# 候选类型

按需使用以下类型，不要求全部覆盖：

- feature：用户可见功能、关键能力缺口、交互改进、集成增强
- security：输入校验、权限、secret、依赖安全、错误处理、安全默认值
- maintenance：重构、清理废弃、目录结构、构建、部署、开发环境
- docs：README、安装、快速开始、架构、贡献指南、示例、FAQ
- test：单测、集成测试、端到端测试、关键路径覆盖
- ci：lint、format、pre-commit、流水线、发布检查
- community：issue / discussion / PR 模板、good first issue、贡献流程、release 规范
- research：需要调研、比较方案、收集用户反馈或形成设计讨论

# 输出格式

## 1. 现状判断与信息来源

用 5-8 条要点说明：

- 项目当前主要能力；
- FOCUS_SCOPE 下最值得推进的方向；
- 最明显的短板、风险或待讨论问题；
- 实际访问到的信息来源；
- 未访问到的信息来源及其影响；
- 是否完成 open issues / discussions 去重。

## 2. 候选事项总表

最多列出 CANDIDATE_COUNT 个候选；不要凑数。

| 优先级 | 标题 | 建议载体 | 类型 | 成熟度 | 工作量 | 证据强度 | 来源 | 推荐理由 |
|---|---|---|---|---|---|---|---|---|

字段要求：

- 优先级：P0 / P1 / P2 / P3
- 建议载体：Issue / Discussion
- 类型：feature / security / maintenance / docs / test / ci / community / research
- 成熟度：可直接创建 / 需补充信息 / 需先讨论
- 工作量：S / M / L
- 证据强度：高 / 中 / 低
- 来源：roadmap / code / issue / discussion / related project / web / inferred
- 推荐理由：1-2 句，说明为什么适合进入 Issue 或 Discussion

## 3. 建议创建的 Issue 草案

只展开“建议载体 = Issue”且“成熟度 = 可直接创建”的候选。

### Issue N：<具体标题>

- 类型 / 优先级 / 工作量 / 建议标签：
- 来源依据：说明具体来自哪个文档、代码位置、issue、discussion 或外部来源；如未完成查重，标注“需人工查重”
- 背景与问题：2-4 句说明为什么需要做、解决什么问题
- 建议范围：列出涉及的模块、目录、配置、文档或流程；明确不在范围内的内容
- 验收标准：列出可客观检查的完成标准，包含必要测试、文档或验证场景
- 风险 / 依赖：仅在有显著兼容性、迁移、权限、安全或前置依赖时填写

## 4. 建议创建或继续的 Discussion 草案

只展开“建议载体 = Discussion”的候选。

### Discussion N：<具体标题>

- 类型 / 优先级 / 建议标签：
- 来源依据：说明具体来自哪个文档、代码位置、issue、discussion、用户反馈或外部来源
- 为什么不直接创建 issue：说明还缺什么共识、信息或决策
- 建议讨论问题：列出 3-5 个需要社区或维护者回答的问题
- 可转为 issue 的条件：说明满足什么条件后可以拆成一个或多个 issue
- 可能的后续 issue：如果已有明显拆分方向，列出候选标题

## 5. 推荐推进顺序

列出最建议优先推进的 3-5 个事项，并说明：

- 哪些应先创建 issue；
- 哪些应先发起或继续 discussion；
- 是否存在依赖关系；
- 如果人手有限，最小可执行组合是什么。

## 6. 可复制标题列表

分别输出可复制列表。

### Issue 标题

- [ ] Issue 标题 1
- [ ] Issue 标题 2

### Discussion 标题

- [ ] Discussion 标题 1
- [ ] Discussion 标题 2

# 写作要求

- 标题要具体，避免“完善 X”“优化 Y”这类空泛标题。
- Issue 的验收标准必须可观测，避免“提升用户体验”“增强稳定性”这类不可验收表述。
- Discussion 的问题要能引导决策，不要只写“大家怎么看”。
- 不要重复创建已有 open issue 或 discussion 已覆盖的事项。
- 不要把需要路线决策或社区共识的事项写成可直接创建 issue。
- 同类项目只能作为参考，必须结合该项目当前阶段判断。
- 输出使用 OUTPUT_LANGUAGE。
```
