# 通用 Issue 提案 Prompt 模板

## 适用场景

在版本节点前（例如 v0.x 收口、公开发布前、社区化前）对一个开源项目做一次系统化的下一阶段 issue 候选盘点。日常的轻量盘点也可基于此模板裁剪使用。

## 建议提供的上下文

- 项目仓库链接（公开仓库或 agent 有权限访问的仓库）
- 下一阶段目标（版本号或阶段名）
- 项目类型（工具库 / Web 系统 / Agent 框架 / 文档型 / CLI 等）
- 期望产出的 issue 数量上限
- agent 是否具备 web 检索与 GitHub API 访问能力

## 使用方式

使用前只需要替换 Prompt 开头变量区中的以下值：

- `REPO_URL`：开源项目仓库链接
- `PROJECT_TYPE`：项目类型，默认填写“软件开发类”，也可按需改为工具库 / Web 系统 / Agent 框架 / 文档型 / CLI / 其他
- `TARGET_STAGE`：下一阶段目标
- `ISSUE_COUNT`：希望产出的 issue 数量上限
- `HAS_WEB_ACCESS`：agent 是否具备 web 检索能力，默认填写“是”

## Prompt

```text
# 变量区：只需要修改本区，其余正文不要手动替换

REPO_URL = <<替换:开源项目链接,例如 https://github.com/org/repo>>
PROJECT_TYPE = 软件开发类
TARGET_STAGE = <<替换:下一阶段目标,例如 v0.2 / v0.3 / 公开发布前 / 社区化阶段>>
ISSUE_COUNT = <<替换:希望产出的 issue 数量上限,例如 10-15;合格不足时宁少勿凑>>
HAS_WEB_ACCESS = 是
OUTPUT_LANGUAGE = 中文
CREATE_ISSUES_DIRECTLY = false

# 任务

你是一名经验丰富的开源项目 maintainer。

请围绕 REPO_URL 指向的开源项目,系统分析项目现状,
并提出该项目在 TARGET_STAGE 阶段适合直接创建的 issue。

这些 issue 应以功能增强为主,
兼顾安全、维护、文档、测试、工程质量和社区协作。

本任务的目标不是提出宽泛想法,
而是筛选出"可以直接派人或派 agent 执行"的 issue。

不要直接创建 GitHub issue,
除非 CREATE_ISSUES_DIRECTLY = true 并经我明确授权。

# 工作原则

1. 不要只基于 README 或表面信息做判断,
   要结合可访问到的代码、文档、roadmap、现有 issues、discussions,
   以及(在 HAS_WEB_ACCESS = 是时)同类项目和外部资料。
2. 每个建议创建的 issue 必须问题明确、边界清晰、可执行、可验收;
   否则归入 discussion 候选或舍弃,不要凑数。
3. 关键不确定性会显著影响 issue 边界、优先级或可执行性时,
   暂停并向我提问;一次只问一个,可多次提问,但不要一次抛多个。
4. 不影响初步判断的不确定性,先明确写出假设再继续分析。
5. 无法访问 REPO_URL 时停止分析并报告原因,不要凭项目名编造内容。

# 分析方法

请按以下顺序执行,
每一步说明实际访问到了什么、未访问到什么以及原因。

## 1. Roadmap 与文档

阅读仓库 README、ROADMAP、TODO、milestone、release plan、CHANGELOG、docs。

重点提取：

- 已明确列出的下一阶段事项;
- 已经承诺但尚未完成的功能;
- roadmap 中粒度过大、需要拆解为可执行 issue 的事项;
- 与 TARGET_STAGE 强相关的任务;
- 过时、重复或不再适合创建 issue 的事项。

## 2. 代码

浏览核心目录,理解模块划分和实现现状。

重点识别：

- 当前能力中明显缺失的部分;
- 代码中 TODO / FIXME / hack / temporary / deprecated 等信号;
- 阻碍后续扩展的架构或抽象问题;
- 错误处理、配置管理、状态管理、权限、日志、测试等方面的不足;
- 部署、安装、开发体验上的明显断点;
- 适合拆成独立 issue 的小型重构或维护任务。

## 3. 已有 open issues 去重(必做)

检索仓库 open 与近期 closed issues。

对每条候选 issue 做去重：

- 如已存在近似 issue,引用其编号并说明本次建议是补充、拆分、放弃,
  还是合并到既有 issue;
- 不要重复创建已有 issue 已涵盖的事项。

## 4. Discussions

阅读现有 discussions,判断：

- 哪些已形成清晰共识、可转化为 issue;
- 哪些仍不成熟、应继续保留为 discussion;
- 哪些用户反馈、使用问题或功能建议可沉淀为 issue;
- 是否有重复 discussion 可合并为同一 issue;
- 是否暗示了文档、示例、安装、部署、配置方面的问题。

## 5. 同类项目与外部资料(HAS_WEB_ACCESS = 是时执行)

参考 1-3 个与该项目在功能、定位或技术栈上相近的开源项目,
以及相关技术文章、最佳实践、安全维护指南。

要求：

- 简要说明每个参考来源为什么相关;
- 不要机械复制对方功能,
  要结合该项目当前阶段判断是否适合借鉴;
- 重点关注功能设计、issue 组织、文档结构、安全实践、
  插件机制、测试体系、社区协作方式;
- 从中提炼适合该项目下一阶段落地的 issue。

如未访问到上述来源,
请明确写出"未执行外部检索",并降低相关 issue 的权重。

# 项目类型适配

根据 PROJECT_TYPE 调整侧重,不强求每类 issue 都覆盖：

- 工具库:侧重 API 设计、向后兼容、版本策略、文档与示例、测试覆盖;
- Web 系统:侧重用户路径、权限、数据一致性、部署运维、UI / UX、可观测性;
- Agent 框架:侧重核心抽象、插件 / 工具机制、可扩展性、可调试性、
  执行隔离、上下文管理、与外部模型的集成;
- 文档型:侧重信息结构、导航、链接一致性、版本管理、可贡献性;
- CLI:侧重命令边界、参数设计、跨平台、错误信息、安装方式。

# Issue 筛选标准

满足以下全部条件,才建议作为 issue;否则归入 discussion 候选或舍弃：

1. 问题明确:一句话能说清楚要解决什么。
2. 边界清晰:知道改哪些模块、文档或流程。
3. 可执行:可以直接派给一个人或一个 agent。
4. 可验收:完成后能客观判断是否达成。
5. 粒度适中:不应大到变成项目,也不应小到没有独立 issue 价值。
6. 与 TARGET_STAGE 相关:不是遥远愿景,也不是无关优化。
7. 不依赖大量未决讨论:需要先讨论方案的归入 discussion 候选。

# Issue 类型建议(按 PROJECT_TYPE 选择性产出)

- feature:用户可见功能新增 / 关键缺口补齐 / 交互改进 / 集成增强
- security:输入校验、权限、secret、依赖安全、错误处理、安全默认值
- maintenance:重构、清理废弃、目录结构、构建 / 部署 / 开发环境
- docs:README、安装、快速开始、架构、贡献指南、示例、FAQ
- test:单测、集成测、关键路径覆盖
- ci:lint、format、pre-commit、流水线
- community:issue / PR 模板、good first issue 标记、release 规范

# 输出格式

## 1. 现状判断与信息来源

5-8 条要点,涵盖：

- 项目当前主要能力;
- 当前阶段最适合增强的方向与最明显的短板;
- 你对 TARGET_STAGE 的理解;
- 本次实际访问到的信息来源,以及未访问到的来源及原因。

## 2. Issue 总表

| 优先级 | 标题 | 类型 | 成熟度 | 工作量 | 来源 | 推荐理由 |
|---|---|---|---|---|---|---|

字段含义：

- 优先级:P0 / P1 / P2 / P3
- 类型:feature / security / maintenance / docs / test / ci / community
- 成熟度:可直接创建 / 需补充少量信息 / 暂不建议
- 工作量:S / M / L
- 来源:roadmap / code / open issues / discussion / related project / web / inferred
- 推荐理由:1-2 句

总数受 ISSUE_COUNT 上限控制;合格不足时宁少勿凑。

## 3. 可直接创建的 Issue 草案

只展开"成熟度 = 可直接创建"的 issue,每个使用以下精简格式：

### Issue N：<标题>

- 类型 / 优先级 / 工作量 / 标签:一行写完
- 来源依据:roadmap / code / open issues / discussion / related project / web / inferred
- 问题陈述:为什么需要这个 issue,要解决什么(2-4 句,结合项目实际)
- 建议范围:涉及的目录、模块、配置、文档;明确不在范围内的内容
- 验收标准:可客观检查的条目;包含必要的测试与验证场景
- 风险或依赖(可选):仅在存在显著风险、兼容性问题或前置依赖时填写

## 4. Discussion 候选

每项一行 + 一句说明：

- 标题;
- 为什么暂不适合 issue、需要先讨论什么、何时可以转为 issue。

## 5. 优先级排序建议

- 列出最建议优先创建的 5 个 issue;
- 说明它们之间是否存在依赖关系;
- 如果人手有限,给出最小可执行组合。

## 6. 可直接复制的标题列表

最后单独输出一个 GitHub task list 形式的标题列表,每行一项,例如：

- [ ] Issue 标题 1
- [ ] Issue 标题 2

# 注意事项

- 不要创建已有 open issues 已涵盖的事项,先去重再下笔。
- 不要把还需要拍板的事项写成 issue,改放 discussion 候选。
- 标题要具体可读,避免"完善 X""优化 Y"这类空泛标题。
- 验收标准要可观测,避免"提升用户体验"这类不可验收的描述。
- 同类项目借鉴不等于复制,要结合该项目当前阶段判断。
```
