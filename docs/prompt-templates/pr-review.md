# 通用 PR Review Prompt 模板

## 适用场景

开源项目收到PR后，让 coding agent 先做一轮结构化审查，帮助 reviewer 更快发现风险、遗漏和需要人工确认的问题。

## 建议提供的上下文

- PR 链接或 diff
- 相关 issue 或任务说明
- 仓库的测试、构建和协作要求
- 希望重点关注的模块或风险点

## 使用方式

适用前提：此模板优先用于 issue 驱动型 PR。
若 PR 没有对应 issue，可在 Prompt 开头变量区将 `ISSUE_ID` 留空，并让 agent 跳过 issue 对齐检查。

使用前只需要替换 Prompt 开头变量区中的以下值：

- `PR_URL`：待评审 PR 的链接，例如 `https://github.com/<org>/<repo>/pull/123`
- `ISSUE_ID`：PR 对应的 issue 编号或链接，例如 `#42`
- `REPO_NAME`：仓库名称，例如 `<repo-name>`

## Prompt

```text
# 变量区：只需要修改本区，其余正文不要手动替换

PR_URL = <<替换:待评审 PR 的链接>>
ISSUE_ID = <<替换:PR 对应的 issue 编号或链接；没有则留空>>
REPO_NAME = <<替换:仓库名称>>

# 任务

你是一名严格但务实的开源项目 PR reviewer。
请评审 PR_URL 指向的 PR 是否可以接受。

重点判断它是否真正解决了 ISSUE_ID 对应的 issue，
并检查代码、文档、测试和仓库规范是否存在问题。

如果 ISSUE_ID 为空，请跳过 issue 对齐检查，
改为根据 PR 标题、描述和实际 diff 判断是否适合合并。

# 评审目标

请给出明确结论：

- 可以接受并合并；
- 需要小修改后再合并；
- 需要较大修改，不建议当前合并；
- 不应合并。

不要只做泛泛评价，要指出具体问题、风险和建议修改点。

# 背景

如果 ISSUE_ID 非空，本 PR 目标是解决 REPO_NAME 仓库中的 issue ISSUE_ID。
请先阅读 issue ISSUE_ID 的描述，再对照 PR 的实际改动判断是否完整解决。
如果 ISSUE_ID 为空，请改为根据 PR 标题、描述和 diff 判断 PR 目标。

重点关注：

1. 是否真正解决了 ISSUE_ID；
2. 是否引入不必要改动；
3. 文档修改是否必要、位置是否合理、命名是否规范；
4. 代码实现是否符合当前项目结构和风格；
5. 是否需要测试，现有测试是否足够；
6. 是否存在边界情况遗漏；
7. 是否会影响现有功能或用户体验；
8. 是否适合直接合并到 main。

# 具体检查项

## 1. Issue 对齐检查

请回答：

- ISSUE_ID 的核心问题是什么？
- PR 是否直接解决了这个问题？
- 是否只解决了一部分？
- 是否存在误解 issue 目标的情况？
- 是否有超出 issue 范围的额外改动？
- 如果有额外改动，这些改动是否合理？

## 2. 代码实现检查

请检查：

- 实现逻辑是否正确；
- 是否覆盖正常输入和异常输入；
- 是否存在边界情况遗漏；
- 是否有硬编码、重复代码或不必要复杂度；
- 是否符合项目现有代码风格；
- 是否影响现有 API、UI 或数据结构；
- 是否存在潜在安全问题；
- 是否存在兼容性问题；
- 是否需要更清晰的错误提示。

## 3. 文档检查

请重点检查：

- 是否确实需要新增或修改文档；
- 文档放置位置是否合适；
- 文档文件名是否符合仓库现有命名习惯；
- 文档内容是否准确、简洁、不过度；
- 是否和 README、docs、roadmap、issue 描述存在冲突；
- 是否出现内部讨论痕迹、临时说明或不适合合并到 main 的内容；
- 是否需要中英文文档同步。

## 4. 测试检查

请检查：

- PR 是否添加或更新了必要测试；
- 如果没有测试，是否合理；
- 是否需要补充单元测试、集成测试或前端测试；
- 测试是否覆盖 issue ISSUE_ID 的关键场景；
- 是否覆盖非法输入、边界输入和正常输入；
- 是否会导致现有测试失败；
- 是否需要手工验证步骤。

## 5. UI / UX 检查

如果 PR 涉及前端或用户输入，请检查：

- 用户看到的提示是否清楚；
- 表单校验是否及时；
- 错误信息是否友好；
- 是否影响现有操作流程；
- 是否有不一致的文案或样式问题。

## 6. 仓库规范检查

请检查：

- 文件路径是否合理；
- 文件命名是否规范；
- 是否引入无关文件、临时文件、构建产物或调试日志；
- commit / PR 改动是否聚焦；
- 是否需要更新 changelog、README 或相关 docs；
- 是否符合开源项目维护习惯。

输出格式：

## 1. 总体结论

结论只能选一个：

- 可以接受并合并；
- 需要小修改后再合并；
- 需要较大修改，不建议当前合并；
- 不应合并。

并用 2-4 句话说明主要理由。

## 2. 是否解决 issue ISSUE_ID

请明确说明：

- ISSUE_ID 的核心要求；
- PR 已完成的部分；
- PR 未完成或可疑的部分；
- 是否存在偏离 issue 的改动。

## 3. 主要问题清单

按严重程度列出：

### Blocker

必须修改，否则不能合并。

### Major

建议修改，否则有明显质量或维护风险。

### Minor

小问题，不一定阻塞合并。

### Nit

命名、格式、表述等细节建议。

如果某类没有问题，请写“无”。

## 4. 文档与命名检查

单独说明：

- 文档修改是否必要；
- 文档位置是否合理；
- 文档命名是否规范；
- 是否建议调整、合并、删除或移动文档。

## 5. 测试与验证建议

请给出：

- 已有测试是否足够；
- 建议补充哪些测试；
- 建议手工验证哪些场景。

## 6. 建议给 PR 作者的修改意见

请整理成可以直接发在 PR review comment 里的文字，语气专业、具体、可执行。

## 7. 最终建议

请给出一句话最终建议：

> 建议合并 / 修改后合并 / 暂缓合并 / 关闭 PR。

# 注意事项

- 不要因为 PR 解决了部分问题就轻易建议合并。
- 不要只看 diff 表面，要结合 issue ISSUE_ID 的目标判断。
- 如果 PR 添加了文档，要特别关注文档位置、命名、内容边界和是否适合 main。
- 如果 PR 修改了校验逻辑，要特别关注边界输入和错误提示。
- 如果发现 PR 做了和 ISSUE_ID 无关的大量改动，请明确指出是否应拆分。
```
