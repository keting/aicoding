# Prompts

本页用于沉淀实验室在 AI coding、Coding Agent、科研协作和工程协作中反复使用的 prompt 模板。

目标不是收集越多越好，而是保留那些可以直接复用、适合教学传播、并且能稳定改善 agent 输出质量的提示词。

## 使用原则

- 优先收录真实项目中反复使用过的 prompt
- 每个 prompt 应说明适用场景、输入材料和期望输出
- prompt 应尽量任务边界清晰，避免只有笼统角色设定
- 对会影响代码、论文或项目决策的 prompt，应要求 agent 明确列出假设、风险和验证方式
- prompt 正文统一放在代码块中，便于在 GitHub 页面一键复制
- 较长 prompt 默认使用 `<details>` 收起，避免页面膨胀
- 代码块内部无法可靠显示颜色或粗体，占位符统一使用 `<<替换:PLACEHOLDER>>` 形式突出标识
- 长 prompt 可以单独放在独立文件中，本页只保留入口和使用说明

## PR Review

适用场景：实验室项目收到 PR 后，让 coding agent 先做一轮结构化审查，帮助 reviewer 更快发现风险、遗漏和需要人工确认的问题。

建议在使用前给 agent 提供以下上下文：

- PR 链接或 diff
- 相关 issue 或任务说明
- 仓库的测试、构建和协作要求
- 希望重点关注的模块或风险点

### 通用 PR Review Prompt 模板

使用前请替换以下高亮占位符：

- <mark><code>&lt;&lt;替换:PR_URL&gt;&gt;</code></mark>：待评审 PR 的链接，
  例如 `https://github.com/keting/half/pull/62`
- <mark><code>&lt;&lt;替换:ISSUE_ID&gt;&gt;</code></mark>：PR 对应的 issue 编号或链接，例如 `#22`
- <mark><code>&lt;&lt;替换:REPO_NAME&gt;&gt;</code></mark>：仓库名称，例如 `HALF`

代码块内部按纯文本渲染，无法可靠显示底色；因此占位符统一使用 `<<替换:...>>` 形式突出标识。

Prompt 正文默认收起。点击下方标题展开后，可使用代码块右上角按钮一键复制。

```text
# 评审目标

请给出明确结论：

- 可以接受并合并；
- 需要小修改后再合并；
- 需要较大修改，不建议当前合并；
- 不应合并。

不要只做泛泛评价，要指出具体问题、风险和建议修改点。

# 背景

本 PR 目标是解决 <<替换:REPO_NAME>> 仓库中的 issue <<替换:ISSUE_ID>>。
请先阅读 issue <<替换:ISSUE_ID>> 的描述，
再对照 PR 的实际改动判断是否完整解决。

重点关注：

1. 是否真正解决了 <<替换:ISSUE_ID>>；
2. 是否引入不必要改动；
3. 文档修改是否必要、位置是否合理、命名是否规范；
4. 代码实现是否符合当前项目结构和风格；
5. 是否需要测试，现有测试是否足够；
6. 是否存在边界情况遗漏；
7. 是否会影响现有功能或用户体验；
8. 是否适合直接合并到 main。

# 具体检查项

## 1. Issue 对齐检查

请回答：

- <<替换:ISSUE_ID>> 的核心问题是什么？
- PR 是否直接解决了这个问题？
- 是否只解决了一部分？
- 是否存在误解 issue 目标的情况？
- 是否有超出 issue 范围的额外改动？
- 如果有额外改动，这些改动是否合理？

## 2. 代码实现检查

请检查：

- 实现逻辑是否正确；
- 是否覆盖正常输入和异常输入；
- 是否存在边界情况遗漏；
- 是否有硬编码、重复代码或不必要复杂度；
- 是否符合项目现有代码风格；
- 是否影响现有 API、UI 或数据结构；
- 是否存在潜在安全问题；
- 是否存在兼容性问题；
- 是否需要更清晰的错误提示。

## 3. 文档检查

请重点检查：

- 是否确实需要新增或修改文档；
- 文档放置位置是否合适；
- 文档文件名是否符合仓库现有命名习惯；
- 文档内容是否准确、简洁、不过度；
- 是否和 README、docs、roadmap、issue 描述存在冲突；
- 是否出现内部讨论痕迹、临时说明或不适合合并到 main 的内容；
- 是否需要中英文文档同步。

## 4. 测试检查

请检查：

- PR 是否添加或更新了必要测试；
- 如果没有测试，是否合理；
- 是否需要补充单元测试、集成测试或前端测试；
- 测试是否覆盖 issue <<替换:ISSUE_ID>> 的关键场景；
- 是否覆盖非法输入、边界输入和正常输入；
- 是否会导致现有测试失败；
- 是否需要手工验证步骤。

## 5. UI / UX 检查

如果 PR 涉及前端或用户输入，请检查：

- 用户看到的提示是否清楚；
- 表单校验是否及时；
- 错误信息是否友好；
- 是否影响现有操作流程；
- 是否有不一致的文案或样式问题。

## 6. 仓库规范检查

请检查：

- 文件路径是否合理；
- 文件命名是否规范；
- 是否引入无关文件、临时文件、构建产物或调试日志；
- commit / PR 改动是否聚焦；
- 是否需要更新 changelog、README 或相关 docs；
- 是否符合开源项目维护习惯。

输出格式：

## 1. 总体结论

结论只能选一个：

- 可以接受并合并；
- 需要小修改后再合并；
- 需要较大修改，不建议当前合并；
- 不应合并。

并用 2-4 句话说明主要理由。

## 2. 是否解决 issue <<替换:ISSUE_ID>>

请明确说明：

- <<替换:ISSUE_ID>> 的核心要求；
- PR 已完成的部分；
- PR 未完成或可疑的部分；
- 是否存在偏离 issue 的改动。

## 3. 主要问题清单

按严重程度列出：

### Blocker

必须修改，否则不能合并。

### Major

建议修改，否则有明显质量或维护风险。

### Minor

小问题，不一定阻塞合并。

### Nit

命名、格式、表述等细节建议。

如果某类没有问题，请写“无”。

## 4. 文档与命名检查

单独说明：

- 文档修改是否必要；
- 文档位置是否合理；
- 文档命名是否规范；
- 是否建议调整、合并、删除或移动文档。

## 5. 测试与验证建议

请给出：

- 已有测试是否足够；
- 建议补充哪些测试；
- 建议手工验证哪些场景。

## 6. 建议给 PR 作者的修改意见

请整理成可以直接发在 PR review comment 里的文字，语气专业、具体、可执行。

## 7. 最终建议

请给出一句话最终建议：

> 建议合并 / 修改后合并 / 暂缓合并 / 关闭 PR。

# 注意事项

- 不要因为 PR 解决了部分问题就轻易建议合并。
- 不要只看 diff 表面，要结合 issue <<替换:ISSUE_ID>> 的目标判断。
- 如果 PR 添加了文档，要特别关注文档位置、命名、内容边界和是否适合 main。
- 如果 PR 修改了校验逻辑，要特别关注边界输入和错误提示。
- 如果发现 PR 做了和 <<替换:ISSUE_ID>> 无关的大量改动，请明确指出是否应拆分。
```

## Issue 拆解

适用场景：把一个模糊想法拆成可执行 issue，尤其适合项目负责人整理 backlog 或给新成员分配任务。

后续可补充：

- 从 Discussion 总结 issue 的 prompt
- 将大任务拆成多个子 issue 的 prompt
- 为调研任务生成完成标准的 prompt

## 代码实现前检查

适用场景：让 agent 在动手改代码前先建立仓库上下文、确认任务边界，降低误改和过度实现的风险。

后续可补充：

- 读仓库并提出实现计划的 prompt
- 检查需求歧义和风险点的 prompt
- 让 agent 明确“不做什么”的 prompt

## 文档整理

适用场景：更新 `aicoding` 这类资源型文档仓库时，让 agent 保持页面职责清晰、避免把资源简单堆叠到同一个页面。

后续可补充：

- 新增资源条目的 prompt
- 合并重复条目的 prompt
- 检查链接、分类和导航一致性的 prompt
